# SUSTech Campus RAG Project Report

## 1. Background and Goal

This project implements Project 1, a SUSTech campus knowledge-base question answering system based on Retrieval-Augmented Generation (RAG). The goal is to collect public official SUSTech information, build a searchable knowledge base, compare retrieval strategies, and provide a local demo system that answers with citations and refuses questions unsupported by the corpus.

The final system covers data ingestion, cleaning, chunking, sparse and dense indexing, hybrid retrieval, reranking, local generation, evidence sufficiency checks, an API service, a Gradio UI, and reproducible evaluation scripts.

## 2. Data Sources

The source registry is defined in `configs/sources.yaml`. It includes official SUSTech websites and public PDF material:

## 3. Data Processing Pipeline

The core pipeline is:

```text
source registry -> crawl/API/PDF fetch -> raw manifest -> parse -> normalize -> chunk -> index -> retrieve -> rerank -> generate
```

Important scripts:

- `scripts/crawl.py`: crawls configured HTML/PDF-capable web sources.
- `scripts/fetch_pdf_sources.py`: discovers and downloads public PDF files from configured directory-style sources.
- `scripts/parse.py`: converts raw HTML/PDF files into normalized documents.
- `scripts/chunk.py`: creates page-aware and overlap-aware chunks.
- `scripts/build_bm25.py`: builds the sparse lexical index.
- `scripts/embed.py`: builds the dense Qdrant index.
- `scripts/generate_eval.py`: creates dev/test/demo evaluation sets.

The PDF parser uses PyMuPDF and preserves page markers. PDF titles are recovered from decoded source URLs so citations show readable titles such as `12-2024级机器人工程专业本科人才培养方案`.

## 4. Model and Retrieval Design

The system supports four retrieval modes:

| mode | description |
| --- | --- |
| `bm25` | sparse lexical retrieval with jieba tokenization |
| `dense` | dense vector retrieval using embedded local Qdrant |
| `hybrid` | reciprocal-rank fusion of BM25 and dense retrieval |
| `hybrid_rerank` | hybrid retrieval followed by local Qwen reranking |

The final local models are:

| component | model |
| --- | --- |
| generator | Qwen2.5-0.5B-Instruct |
| embedding | Qwen3-Embedding-0.6B |
| reranker | Qwen3-Reranker-0.6B |

The generator is served through a local OpenAI-compatible endpoint on port `8000`. The API service calls the same endpoint as it would call a hosted OpenAI-compatible model, keeping the demo fully local.

The Qwen3 reranker is implemented with Qwen's yes/no causal scoring formulation. For stability on this relatively small course-project corpus, the reranker score is blended with the original hybrid rank prior.

## 5. Answer Generation and Refusal

For answerable questions, the system builds a citation-aware context from the top retrieved chunks and asks the local LLM to answer only from official evidence. Each response contains:

- generated answer
- evidence status
- source citations
- retrieval mode
- trace ID
- retrieval trace

For unsupported questions, the system returns a refusal instead of hallucinating. The evidence sufficiency check is intentionally strict about concrete qualifiers such as years and campus names, while ignoring template words such as "资料", "相关内容", or "教学工作部" so PDF curriculum questions are not falsely rejected.

Example unsupported question:

```text
南方科技大学是否提供量子传送门预约服务？
```

Expected response:

```text
当前知识库没有检索到足够直接证据，无法根据现有官方资料回答该问题。
```

## 8. Reproducibility

A full rebuild can be run with:

```bash
cd /root/RAG_proj/sustech-rag
bash scripts/run_pipeline_core.sh
python scripts/fetch_pdf_sources.py
python scripts/parse.py
python scripts/chunk.py
python scripts/build_bm25.py
python scripts/embed.py
python scripts/generate_eval.py
python scripts/evaluate.py --split dev
python scripts/evaluate.py --split test
python scripts/evaluate_generation.py --split test --mode hybrid_rerank --use-llm
python scripts/report_summary.py
```

Embedded Qdrant uses a local file lock, so direct Python indexing/evaluation jobs should be run sequentially unless retrieval is mediated by the API service.

## 9. Limitations and Future Work

- The corpus is large enough for the course project but still not exhaustive across every SUSTech department.
- Some official pages contain navigation text, which can affect sparse retrieval. The evaluation generator filters obvious navigation-heavy chunks, and future work could add more aggressive boilerplate removal.
- The final generator uses Qwen2.5-0.5B-Instruct for AutoDL resource compatibility. A larger model such as Qwen2.5-7B-Instruct could improve answer fluency if GPU memory allows.
- The current reranker is conservative. A larger reranker or supervised calibration on a manually labeled set could improve test-set reranking.

## 10. Conclusion

The project delivers a complete local RAG system for SUSTech campus information. It includes official-source ingestion, PDF support, reproducible indexing, multiple retrieval modes, local LLM generation, citation display, refusal handling, automated evaluation, and a working API/UI demo.
